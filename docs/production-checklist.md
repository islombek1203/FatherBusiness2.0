## Security checklist

- [x] **Passwords**: bcrypt hashing; the seed script's bootstrap Admin password is generated with a CSPRNG (`node:crypto` `randomInt`), not `Math.random()`.
- [x] **Timing side-channels**: login always runs `bcrypt.compare` (against a dummy hash for nonexistent/inactive users), so response timing can't be used to enumerate valid emails.
- [x] **Sessions**: real, server-side, database-backed sessions (not JWT-in-cookie) — see `architecture.md`'s Auth model section. Instantly revocable; invalidated on logout, password change, and database restore.
- [x] **CSRF**: `SameSite=Lax` session cookie (Auth.js default) plus an explicit `Sec-Fetch-Site` same-origin check (`src/lib/require-same-origin.ts`) on the two most sensitive endpoints, backup and restore.
- [x] **Authorization**: every mutating server action calls `requireRole()` server-side; route-level redirects (`proxy.ts`) are a UX convenience only, never the actual boundary. Verified by reading every write path, not just the gated routes.
- [x] **Parameterized queries**: Prisma ORM exclusively — no `$queryRaw`/`$executeRaw`/raw SQL anywhere in application code (verified by grep; the only hits are in the generated Prisma client's own type definitions).
- [x] **Secrets**: `.env` is gitignored; only `.env.example` (placeholder values) is committed. Verified nothing under a `.env*` pattern is tracked in git.
- [x] **Login rate limiting**: 5 failed attempts / 15 minutes per email; only failures count (concurrent legitimate logins from the same account are never blocked — this was a real bug found and fixed during Phase 2 e2e testing).
- [x] **Security headers**: set from `next.config.ts`'s `headers()` (works standalone, without depending on the reverse proxy) and mirrored in `Caddyfile` for the Docker deployment:
  - `Content-Security-Policy` — see the caveat below, this one is not as strong as intended.
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Permissions-Policy: camera=(), microphone=(), geolocation=()`
  - `X-Powered-By` header removed (`poweredByHeader: false`)
  - Caddy also auto-adds `Strict-Transport-Security` once a real TLS certificate is issued (not applicable to `localhost`/plain-HTTP dev).

### Known limitation: CSP has no script nonce

The Next.js-documented approach for a strict CSP (`script-src` restricted to `'self'` + a per-request nonce, via `proxy.ts`) was implemented first, but **response headers set from `proxy.ts` do not reach the client in this Next.js 16.2.10 build** — verified with a minimal reproduction (a proxy that does nothing but `response.headers.set(...)`) under both `next dev` and `next start`; only `next.config.js`'s `headers()` output was observed reaching the browser.

The shipped CSP is therefore static (no nonce), which requires `script-src 'self' 'unsafe-inline'` — the App Router injects inline `<script>` tags carrying the RSC hydration payload on every page load, and without a nonce there's no way to allowlist only those. This weakens the policy's protection against injected inline `<script>` content specifically (external script loading, `eval`, plugins/objects, and clickjacking are still blocked).

Mitigating factors: the app has no `dangerouslySetInnerHTML` anywhere (verified by grep) and never renders user-supplied input as raw HTML — React's default text-node escaping applies everywhere data is displayed. Residual risk is low but not zero.

**Before a security-sensitive production launch**: re-test whether a newer Next.js release fixes proxy-set response header propagation, and switch back to the nonce-based policy (the code pattern is documented in `node_modules/next/dist/docs/01-app/02-guides/content-security-policy.md` if you want to re-attempt it) to drop `'unsafe-inline'` from `script-src`.

## Deployment readiness

- [ ] **Docker Compose has not been build-tested on this development machine** (no Docker installed here — see `PLAN.md` Phase 1). `Dockerfile`, `docker-compose.yml`, `docker-entrypoint.sh`, and `Caddyfile` are all written as required deliverables and believed correct, but **must** be verified with a real `docker compose up --build` before production use. See `deployment.md`'s verification section for the specific things to check (Postgres client/server version match for `pg_dump`/`pg_restore`, Caddy TLS issuance against a real domain).
- [ ] **`next/font/google` requires outbound network access at build time.** This was observed to be flaky (intermittent fetch failures, succeeding on retry) even in an environment with working general internet access. If your production build environment has restricted/no egress, the build will fail outright — either ensure egress to Google Fonts is allowed during `docker compose up --build`/CI, or migrate to `next/font/local` with vendored font files (the PDF export already does this pattern for its own font — see `public/fonts/`).
- [ ] Generate real, unique `AUTH_SECRET` and `POSTGRES_PASSWORD` values for production — never reuse the placeholder/dev values from `.env.example`.
- [ ] Confirm DNS for `DOMAIN` resolves to the VPS before first `docker compose up`, so Caddy's ACME HTTP challenge succeeds.
- [ ] Set up an external backup schedule (see `backup-restore.md`) — the app does not back up automatically on its own.

## Explicitly out of scope (per the fixed build spec)

Barcode/QR/camera scanning, multi-store/warehouse support, cloud sync, analytics, AI features, external API integrations. These were deliberately left as design seams, not implemented — see `PLAN.md`'s "Future-proofing" section.
